# Подготовка и развертывание Linux VM

> **Публичный учебный пример.** Все названия, домены, IP-адреса, роли и процессы в этом репозитории вымышлены.  
> Инструкция показывает общий подход и не должна применяться в production без адаптации, проверки безопасности и тестирования.

## Назначение

Этот материал описывает жизненный цикл обычной виртуальной Linux-машины:

1. сбор требований;
2. выделение ресурсов, сети и DNS;
3. развертывание из шаблона;
4. первичная настройка Linux;
5. опциональное присоединение к Active Directory;
6. подключение мониторинга, резервного копирования и учета;
7. приемочные проверки и передача владельцу.

Пример рассчитан на виртуальную инфраструктуру класса VMware vSphere, но общий процесс применим и к другим гипервизорам.

## Границы документа

Документ не содержит:

- реальных адресов, доменов, имен сотрудников и внутренних групп;
- паролей, токенов, ключей и служебных учетных записей;
- точных правил межсетевого экрана;
- внутренней структуры CMDB, IPAM, DNS и Active Directory;
- конкретных требований к антивирусу, SIEM, мониторингу или резервному копированию.

## Роли

| Роль | Ответственность |
|---|---|
| Заказчик | Формулирует назначение сервера, требования к ресурсам, доступу и SLA |
| Команда виртуализации | Создает VM, подключает диски и виртуальные сети |
| Команда сети/DNS/IPAM | Выделяет адреса, шлюзы, DNS-записи и маршруты |
| Команда идентификации | Делегирует права на ввод в домен и определяет OU/политики |
| Linux Operations | Настраивает ОС, доступ, обновления и базовую защиту |
| Security | Проверяет соответствие политике безопасности |
| Monitoring/Backup | Подключает наблюдаемость и резервное копирование |
| Владелец сервиса | Выполняет приемку и подтверждает готовность |

Названия отдельных людей в рабочих инструкциях использовать не следует. Ответственность должна быть привязана к роли, очереди или группе.

## Входные данные

До начала развертывания должны быть известны:

| Параметр | Пример |
|---|---|
| Имя VM | `linux-vm-01` |
| FQDN | `linux-vm-01.corp.example.com` |
| Назначение | сервер приложения |
| ОС | поддерживаемая Ubuntu LTS или RHEL-совместимая ОС |
| vCPU / RAM | `4 vCPU / 8 GiB` |
| Диски и точки монтирования | системный диск и отдельный диск данных |
| Сетевые сегменты | production, backup, management |
| IP / prefix / gateway | адреса из IPAM |
| DNS и search domain | корпоративные DNS-серверы |
| NTP | корпоративные источники времени |
| Доступ | административные группы и сервисные учетные записи |
| Active Directory | домен, OU и способ интеграции |
| Backup | RPO, RTO и перечень данных |
| Monitoring | хост, шаблон, теги и ответственные |
| Maintenance | окно обслуживания и правила перезагрузки |

Если обязательные данные отсутствуют, запрос возвращается на уточнение. Значения нельзя угадывать.

### 1. Подготовка

Перед созданием VM:

- подтвердить владельца и назначение;
- проверить квоты, кластер, datastore и политику размещения;
- зарезервировать IP-адреса;
- определить прямые и обратные DNS-записи;
- согласовать правила сетевого доступа;
- определить группы администраторов;
- определить требования к резервному копированию и мониторингу;
- при необходимости подготовить объект компьютера или OU в AD.

Предварительное создание объекта компьютера в AD — это организационная опция, а не универсальное техническое требование. Оно необходимо только тогда, когда так устроены делегирование прав и процесс управления объектами.

### 2. Развертывание VM

Предпочтительный способ — поддерживаемый и регулярно обновляемый шаблон VM:

- шаблон содержит актуальную ОС и guest tools;
- отсутствуют уникальные machine-id, SSH host keys и другие клонируемые идентификаторы;
- включен `cloud-init` или механизм guest customization;
- базовая конфигурация описана как код;
- версия и дата сборки шаблона известны;
- шаблон проходит регулярное сканирование.

Установка с ISO оправдана, когда отсутствует пригодный шаблон или требуется нестандартная разметка. Генерация специальных ISO внутренним скриптом не должна быть единственным способом развертывания.

При создании VM задаются:

- имя и папка;
- кластер или resource pool;
- datastore;
- тип гостевой ОС;
- CPU и RAM;
- диски;
- виртуальные сетевые адаптеры;
- сети/port groups;
- теги владельца, среды и сервиса.

Не храните учетные данные vCenter в репозитории. Используйте secret store, переменные CI/CD с ограниченным доступом или интерактивную аутентификацию.

### 3. Имя хоста

Пример:

```bash
sudo hostnamectl set-hostname linux-vm-01.corp.example.com
hostnamectl
hostname --fqdn
```

VM, DNS, CMDB и ОС должны использовать согласованное имя. Требование к регистру нужно определить заранее и применять единообразно.

Не добавляйте FQDN в `/etc/hosts` как замену корректному DNS. Локальные записи допустимы только при явной необходимости и должны быть документированы.

### 4. Сеть: Ubuntu и Netplan

Сначала определите реальное имя интерфейса:

```bash
ip -brief link
ip -brief address
```

Пример статической настройки приведен в [`examples/netplan-single-nic.yaml`](examples/netplan-single-nic.yaml).

Перед применением:

```bash
sudo netplan generate
sudo netplan try
sudo netplan apply
netplan status --all
```

`netplan try` особенно важен при удаленной настройке: при потере соединения конфигурация может быть автоматически отменена.

Для нескольких интерфейсов:

- используйте один основной default route, если нет специальной схемы;
- для backup/management сетей задавайте явные маршруты;
- при нескольких шлюзах учитывайте метрики;
- при асимметричной маршрутизации применяйте policy-based routing;
- не используйте `on-link: true`, если шлюз действительно находится в подсети интерфейса.

Пример двух интерфейсов находится в [`examples/netplan-dual-nic.yaml`](examples/netplan-dual-nic.yaml).

### 5. Сеть: RHEL-совместимые системы

Для современных RHEL-совместимых систем используйте NetworkManager через `nmcli`, `nmtui`, `nmstate` или системную роль. Не переносите в новую инструкцию ручное редактирование legacy-файлов `ifcfg-*` как основной способ.

Пример:

```bash
sudo bash examples/rhel-network-nmcli.sh
```

Перед запуском замените значения в начале файла.

Проверка:

```bash
nmcli device status
nmcli connection show --active
ip -brief address
ip route
resolvectl status 2>/dev/null || true
```

### 6. Время и DNS

Kerberos и доменная аутентификация чувствительны к времени и DNS.

```bash
timedatectl
chronyc tracking 2>/dev/null || true
getent hosts linux-vm-01.corp.example.com
getent hosts 192.0.2.20
```

Проверяйте не только `ping`: ICMP может быть запрещен. Дополнительно проверяйте разрешение имен, маршруты и доступность требуемых TCP/UDP-портов.

### 7. Обновления

Обновления выполняются в согласованное окно обслуживания.

Ubuntu:

```bash
sudo apt update
sudo apt upgrade
```

RHEL-совместимая система:

```bash
sudo dnf upgrade --refresh
```

Нельзя универсально отвечать «оставить локальную версию» или «заменить конфигурацию поставщика» во всех диалогах пакетного менеджера. Изменения конфигурационных файлов нужно сравнить и объединить с учетом локальных настроек.

Перезагрузка выполняется при фактической необходимости, а не после каждого шага:

```bash
sudo systemctl reboot
```

Legacy-команды вроде `init 6` не используются в новой инструкции.

### 8. Базовая безопасность

Минимальный набор:

- запрет прямого SSH-входа под `root`;
- аутентификация по ключам, где это применимо;
- персональные административные учетные записи;
- минимально необходимые sudo-права;
- firewall и SELinux/AppArmor согласно политике;
- журналирование административных действий;
- автоматическая установка критических обновлений либо управляемый patch cycle;
- сканирование уязвимостей;
- синхронизация времени;
- защита секретов.

Не задавайте «стандартный общий пароль root». Общие пароли и широкие правила вида `NOPASSWD: ALL` создают высокий риск.

Пример безопасного фрагмента sudoers:

```sudoers
%linux-admins ALL=(ALL:ALL) ALL
```

Изменения выполняются через отдельный файл и проверяются:

```bash
sudo visudo -f /etc/sudoers.d/linux-admins
sudo visudo -c
```

Если группе нужен беспарольный запуск, перечисляйте конкретные команды, а не `ALL`.

### 9. Присоединение к Active Directory

Для обычной аутентификации пользователей на Linux предпочтителен `realmd` + SSSD. Winbind выбирается, когда он нужен для Samba-сервисов или совместимости с существующей архитектурой.

#### Предварительные проверки

```bash
hostname --fqdn
timedatectl
realm discover corp.example.com
```

Требования:

- Linux использует DNS, способный разрешать AD-зону и SRV-записи;
- время синхронизировано с доменной инфраструктурой;
- hostname и DNS уникальны и согласованы;
- учетная запись имеет делегированные права на нужную OU;
- сетевые порты открыты;
- секреты не указаны в командной строке и не записываются в Git.

#### Ubuntu: SSSD

Установите пакеты, рекомендованные для вашей поддерживаемой версии Ubuntu:

```bash
sudo apt update
sudo apt install realmd sssd sssd-tools libnss-sss libpam-sss adcli
```

Присоединение:

```bash
sudo realm discover corp.example.com
sudo realm join corp.example.com -U ad-join-user
```

Команда запросит пароль интерактивно.

Проверка:

```bash
realm list
getent passwd 'user@corp.example.com'
id 'user@corp.example.com'
sssctl domain-status corp.example.com
```

Создание домашних каталогов и правила доступа должны настраиваться согласно политике организации, а не включаться безусловно для всех доменных пользователей.

#### RHEL-совместимая система: SSSD

```bash
sudo dnf install realmd sssd adcli oddjob oddjob-mkhomedir krb5-workstation
sudo realm discover corp.example.com
sudo realm join corp.example.com -U ad-join-user
```

Проверка:

```bash
realm list
id 'user@corp.example.com'
sssctl domain-status corp.example.com
```

На актуальных RHEL-подобных системах не используйте старый `authconfig`. Интеграция через `realm join` и SSSD настраивает необходимые компоненты, а ручные изменения PAM/NSS должны выполняться только по документации поддерживаемой версии.

#### Когда нужен Winbind

Если сервер является Samba member server или архитектура требует Winbind:

```bash
sudo realm join \
  --membership-software=samba \
  --client-software=winbind \
  corp.example.com
```

Пакеты и дополнительные настройки зависят от дистрибутива. Не смешивайте SSSD и Winbind без обоснованного проекта миграции.

### 10. Доступ

Разрешайте вход только нужным группам.

Пример для `realmd`:

```bash
sudo realm deny --all
sudo realm permit -g 'linux-admins@corp.example.com'
```

Перед применением проверьте формат имен групп в вашей AD-схеме и сохраните локальный аварийный доступ.

### 11. Monitoring, Backup и Security

После базовой настройки:

- зарегистрировать узел в monitoring;
- проверить поступление метрик и алертов;
- установить агент безопасности одобренным способом;
- проверить, что узел появился в соответствующей консоли;
- настроить backup согласно RPO/RTO;
- выполнить тестовое резервное копирование;
- зафиксировать исключения и владельцев;
- исключить секреты из inventory и логов.

Агенты и playbook должны получать адреса серверов, токены и сертификаты из переменных или secret store.

### 12. Автоматизация

Ручной процесс допустим для разовой диагностики, но не как основной путь массового развертывания.

Рекомендуемое разделение:

- **Packer/Image Builder** — сборка шаблона;
- **Terraform или Ansible** — создание VM;
- **cloud-init** — первоначальная конфигурация гостя;
- **Ansible** — hardening, пакеты, домен, monitoring и backup;
- **CI** — lint, проверка YAML, secret scanning и тестовый deploy;
- **CMDB/IPAM API** — регистрация и получение адресов.

Пример cloud-init находится в [`examples/cloud-init.yaml`](examples/cloud-init.yaml). Он не содержит паролей и использует публичный SSH-ключ-заглушку.

Автоматизация должна быть идемпотентной: повторный запуск не создает дубликаты, не ломает сеть и не перезаписывает рабочую конфигурацию без контроля.

## Приемочные проверки

### Система

```bash
hostnamectl
cat /etc/os-release
systemctl --failed
journalctl -p err -b --no-pager
```

### Сеть

```bash
ip -brief address
ip route
getent hosts linux-vm-01.corp.example.com
```

Проверяйте:

- ожидаемый IP на каждом интерфейсе;
- ровно те default routes, которые предусмотрены дизайном;
- прямое и обратное DNS-разрешение;
- доступ к репозиториям, DNS, NTP и целевым сервисам;
- отсутствие неожиданных открытых портов.

### Домен

```bash
realm list
id 'user@corp.example.com'
```

### Monitoring и Backup

- узел имеет состояние `up/healthy`;
- поступают CPU, RAM, filesystem и service checks;
- тестовый alert доставлен;
- backup job успешен;
- восстановление проверено или запланировано.

### Безопасность

- прямой root login отключен;
- доступ имеют только согласованные группы;
- sudo-политика проверена;
- секретов нет в shell history и репозитории;
- критических ошибок и незакрытых исключений нет.

## Критерии готовности

Сервер считается готовым, когда:

- VM создана в правильном кластере и сети;
- hostname, IP, DNS и время настроены;
- ОС обновлена в рамках утвержденного окна;
- базовая защита применена;
- доменная интеграция работает, если она нужна;
- доступ выдан по принципу минимальных привилегий;
- monitoring и backup проверены;
- запись в реестре активов заполнена;
- владелец получил сведения о доступе, обслуживании и поддержке.

## Откат

План отката определяется до изменений.

Возможные действия:

- вернуть предыдущую сетевую конфигурацию через консоль гипервизора;
- выполнить `realm leave corp.example.com`;
- восстановить проверенную конфигурацию из Git/backup;
- удалить неиспользуемые DNS- и AD-объекты;
- удалить тестовую VM после подтверждения отсутствия данных;
- восстановить snapshot только в пределах утвержденной политики.

Snapshot VM не является полноценной резервной копией и не должен храниться бесконтрольно.

## Публичные тестовые значения

В примерах используются зарезервированные диапазоны документации:

- `192.0.2.0/24`;
- `198.51.100.0/24`;
- `203.0.113.0/24`;
- домен `example.com`.

Замените их значениями из своей IPAM/DNS-системы.
